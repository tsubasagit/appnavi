# AppNavi 基本構想案

## 1. ビジョン & キャッチコピー

**「データは個人のもの、プラットフォームはみんなのもの」**

### Vision

ITの専門知識を持たないすべての人々が、自身のデータを自由に活用し、自らの手で業務課題を解決できる社会を実現する。

### Core Philosophy

#### Data Sovereignty (データの主権)

データはユーザー自身（個人や企業）の資産であり、プラットフォームが囲い込むものではない。ExcelやGoogleスプレッドシートという「ユーザーの手元」にあるデータを正とし、アプリはその「窓」としての役割に徹する。

#### Platform Commons (プラットフォームの共有地化)

アプリを動かす基盤やテンプレート、ノウハウはオープンソース（OSS）として共有され、誰でも低コストで利用・改良できる公共財とする。

## 2. ターゲットユーザーと課題

### メインターゲット: 「現場のリーダー」

- **中小企業の経営者**: IT投資予算がなく、Excelでの手作業に限界を感じているが、高額なSaaS導入には踏み切れない。
- **自治体職員**: DX推進を求められているが、セキュリティ要件や予算制約が厳しく、ベンダーロックインを避けたい。
- **NPO・非営利団体**: 活動資金を現場に回したいため、管理ツールにはコストをかけられない。

### 解決する課題 (Pain Points)

- **「作る」ハードル**: No-Codeツールですら「画面設計」や「データベース設計」が必要で難しすぎる。
- **「壊れる」恐怖**: アプリ導入で、慣れ親しんだExcelデータが消えたり、複雑になったりするのが怖い。
- **「コスト」の壁**: サーバー維持費や月額利用料が負担。

## 3. 中小企業DXの第一歩としての活用方針 (DX Strategy for SMEs)

AppNaviは、ITベンダーに丸投げするのではなく、「自社で小さく始める」ことをDXの第一歩と定義します。

### 「デジタル化」から「DX」への架け橋

組織の成熟度に合わせて、以下の3ステップで変革を支援します。

#### Step 1: デジタイゼーション (Digitization - 守りのデジタル化)

**方針**: 「紙と電話」や「個人のPCに眠るExcel」を、まずはクラウド上の共有データに置き換える。

**AppNaviの役割**: 既存のExcelファイルをそのままアプリの裏側として使うことで、業務フローを大きく変えずに、データ収集のデジタル化（クラウド化）を即座に実現します。

#### Step 2: デジタライゼーション (Digitalization - 攻めのデジタル化)

**方針**: デジタル化されたデータを活用し、業務プロセス自体を効率化・自動化する。

**AppNaviの役割**: 自動集計グラフやリアルタイム共有により、毎月の「集計作業」や「報告会議」のための資料作成時間をゼロにします。

#### Step 3: デジタルトランスフォーメーション (DX - 経営の変革)

**方針**: データに基づいて経営判断を行い、ビジネスモデルや組織文化を変革する。

**AppNaviの役割**: 「方針」タブで定義したKPI（成果指標）をダッシュボードで常時モニタリングし、「勘と経験」に頼らないデータドリブンな経営判断を日常化させます。

### 「Small Wins (小さな成功体験)」の重視

いきなり全社規模の基幹システムを導入するのではなく、現場の小さな課題（日報、備品管理、アンケートなど）から成功体験を積み重ねます。「自分たちでもITで課題解決ができた」という自信（デジタル・エフィカシー）こそが、中小企業のDXを推進する最大のエンジンとなります。

## 4. ソリューションの特徴 (3つのNO + AI)

AppNaviは、従来の「開発ツール」ではなく、**「意思決定と可視化のアシスタント」**として位置づける。

### ① No Code (コード不要・設計不要)

- **「作る」から「選ぶ」へ**: ユーザーはドラッグ＆ドロップで画面を作る必要がない。「テンプレートから選ぶ」か「AIに任せる」だけでアプリが完成する。
- **方針（Policy）ドリブン**: 「何のためのアプリか？」を入力するだけで、AIが最適なデータ構造とグラフを提案する。

### ② No Cost & No Server (運用コストゼロ・サーバーレス)

- **BYOD (Bring Your Own Data)**: データベースはユーザー自身のGoogleスプレッドシートやExcelファイル。
- **Serverless Architecture**: FirebaseやGitHub Pages等の無料枠/安価なインフラを活用し、サーバー管理の手間とコストを排除。

### ③ No Fear (恐怖感の払拭)

- **一方通行の同期**: アプリ側からの操作は、元の重要なExcel数式を破壊しないよう安全に設計。
- **いつでも戻れる**: データは手元にあるため、AppNaviの使用をやめてもデータは残る（ベンダーロックインなし）。

## 5. プロダクト構成とUXフロー

ユーザー体験は以下の4ステップで完結する。

### STEP 0: 方針策定 (Policy & Strategy)

**機能**: 「方針」タブの実装。

**UX**: ユーザーは「現状の課題」「解決後のゴール」「成果指標（KPI）」を言語化して入力する。

**AI活用**: 入力された方針をシステムプロンプト（核となる指示書）として保存。以降の全工程でAIがこれを参照し、「この方針なら、このグラフが必要です」と提案を行う。

### STEP 1: データ連携 (Connect)

**機能**: Excel/CSVアップロード、Googleスプレッドシート連携。

**UX**: ファイルをドラッグするだけ。データ型（日付、金額など）は自動判定。

**安心設計**: 「元のファイルは変更されません」というメッセージを常時表示。

### STEP 2: デザイン生成 (Generate UI)

**機能**:
- **リスト一覧型 / カード型 / カレンダー型**: 用途に合わせて大枠を選択。
- **テンプレート選択**: 「在庫管理」「日報」「顧客管理」などの定番型を選択。
- **AIオートデザイン**: 「スマホで見やすい報告書を作って」等の指示で自動生成。

**UX**: 細かい配置作業はさせない。プレビューを見ながらスイッチで表示/非表示を切り替えるだけ。

### STEP 3: 運用・分析 (Dashboard & Insight)

**機能**: グラフ自動生成、データ共有設定。

**UX**: STEP 0で設定した「成果指標」に基づき、達成度を可視化するグラフが自動で配置される。

**セキュリティ**: 「社内限定（Googleログイン必須）」や「パスワード保護」をワンクリックで設定。

## 6. 技術戦略とセキュリティ

### フロントエンド
- React + Tailwind CSS (軽量・高速・レスポンシブ)

### インフラ
- Firebase (Authentication, Hosting, Firestore)

### AIエンジン
- Gemini API (コンテキスト理解、コード生成、データ分析)

### データフロー

- **パブリックデータ**: Firestore等でキャッシュ
- **プライベートデータ**: ユーザーのGoogle Drive/ローカルストレージと直接通信（AppNaviサーバーにデータを保存しないオプションも検討）

---

## 7. ロードマップ

### フェーズ1（MVP）
スプレッドシート連携と基本テンプレート（リスト/カード）の実装。

### フェーズ2（AI統合）
「方針」タブの実装と、それに基づくUI/グラフのAI自動生成機能。

### フェーズ3（エコシステム）
ユーザーが作成したテンプレートを（個人情報を抜いた状態で）「みんなのプラットフォーム」として共有できるマーケットプレイス機能。
